# 《今天干点啥》最终定稿方案

版本：Final 1.0  
产出日期：2026-03-23  
适用对象：产品决策、版本规划、研发评审、运营对齐

---

## 0. 这份文档解决什么问题

这不是一份“点子合集”，也不是“功能许愿清单”。

这份文档要解决的，是下面 5 个关键问题：

1. 《今天干点啥》最终到底要做成什么产品
2. 哪些功能是核心，哪些功能现在不能乱做
3. 恋人系统怎样从附属功能变成产品灵魂
4. 社区、服务、记忆系统怎么围绕主线形成闭环
5. 接下来 3 个阶段应该先做什么，怎么走成黑马

这份“最终定稿方案”融合了三版方案的优点：

1. 保留第一版的玩法想象力、互动传播力、情侣链路设计
2. 吸收第二版对场景扩展、自定义场景、内容与商业化的系统化思路
3. 采用第三版最关键的判断：聚焦主线、减少发散、围绕“关系驱动的日常决策”建立护城河

---

## 1. 最终结论

### 1.1 产品最终定义

《今天干点啥》最终不应该只是一个“随机推荐器”，也不应该只是一个“情侣互动小程序”。

它最有机会做成的是：

**一款围绕“今天做什么”展开，以“关系协作”为核心差异化，以“灵感决策 + 行动触发 + 共同记忆”为完整闭环的日常生活小程序。**

更通俗一点说：

**它帮助用户在一天中更轻松地决定吃什么、玩什么、拍什么、约什么，并且把这些决定变成与恋人、搭子、朋友之间真实发生的互动。**

如果做对了，它不是“今天吃点啥”的扩展版，而是：

**年轻人的日常生活灵感与关系协作操作系统。**

---

### 1.2 这个产品成为黑马的核心，不在“功能多”，而在“关系驱动”

市面上从来不缺：

1. 推荐吃什么的工具
2. 推荐去哪玩的内容平台
3. 能发帖子、写攻略的社区
4. 情侣签到、情侣空间、情侣纪念日工具
5. 能跳转外卖、团购、地图、购票的平台

真正稀缺的是：

**把“做决定”这件事，变成关系中的互动。**

这件事一旦成立，产品价值就会从“帮我选一个东西”，升级成：

1. 帮我发起行动
2. 帮对方给出回应
3. 帮我们一起完成
4. 帮我们留下记忆

这就是《今天干点啥》最强的差异化，也是它最像黑马的地方。

## 2. 先看现状：你现在已经有什么

基于当前小程序代码结构，你已经具备以下基础盘：

1. 首页和模块入口
2. `今天吃点啥`
3. 发布帖子
4. 发现页内容流
5. 点赞、评论、收藏、分享
6. 登录、注册、找回密码、个人资料
7. 恋人系统雏形
8. 美团跳转、菜谱/图片生成、相关文章联动的基础思路

这说明你并不是从 0 做产品定位，而是已经有了：

**“一个生活决策工具 + 一个内容社区雏形 + 一个关系系统起点”**

这非常重要。

因为这决定了你接下来的正确方向不是“继续堆页面”，而是：

**把现有这些能力，沿着同一条主线串起来。**

---

## 3. 当前最大问题是什么

目前最大的问题，不是功能少，而是产品主线不够清晰。

核心问题有 4 个：

1. 各模块更像平行入口，没有形成“今天做什么”的统一主线
2. 恋人系统目前更像装饰功能，还不是增长引擎
3. `吃 / 玩 / 拍 / 笑 / 做 / 聊` 容易越做越散
4. 社区、服务、记忆还没有和核心场景形成闭环

所以接下来最重要的，不是“想到一个功能就做一个”，而是先明确：

**这个产品最终是一款什么产品，它的第一增长引擎是什么，它的独特之处是什么。**

我的最终判断是：

**《今天干点啥》应该是一款围绕日常决策展开、以关系协作为核心差异化、以社区与本地生活服务为增长飞轮的生活方式小程序。**

---

## 4. 为什么这条方向成立

### 4.1 小程序天然适合做“高频轻决策”

微信小程序天然适合：

1. 高频打开
2. 轻量操作
3. 场景即用即走
4. 基于微信关系链传播

而《今天干点啥》的核心需求恰好就是：

1. 今天吃什么
2. 今天下班后玩什么
3. 今天拍点什么
4. 今天要不要约谁一起
5. 今天想不想和恋人互动一下

这些都不是重度工具需求，而是典型的**高频轻决策需求**。

这和小程序生态非常匹配。

---

### 4.2 年轻用户越来越重视“情绪价值”

今天的年轻用户已经不只是想解决问题，还想得到：

1. 被理解
2. 被回应
3. 被陪伴
4. 仪式感
5. 惊喜感
6. 可分享的生活瞬间

所以如果《今天干点啥》只做“推荐吃什么”，价值很容易见顶。

但如果它能把“吃什么、去哪玩、拍什么”变成关系里的互动动作，它的价值就会上一个台阶。

用户用的就不是一个工具，而是一种日常陪伴方式。

---

### 4.3 “玩点啥”和“拍点啥”是真实需求，不是伪扩展

用户真正高频的不是“深度旅游”，而是：

1. 今晚下班后去哪
2. 周末半天去干嘛
3. 预算不高能玩什么
4. 下雨天适合去哪里
5. 和恋人适合做什么
6. 今天想发朋友圈但不知道拍什么

所以“今天玩点啥”“今天拍点啥”是成立的。

但要注意：

**它们不是做成大而全平台，而是做成“轻量、即时、可执行”的生活灵感决策器。**

---

### 4.4 真正稀缺的是“从灵感到行动”的关系链路

今天内容种草平台已经很强了，服务平台也很成熟了。

你不应该跟它们正面竞争。

你真正该做的是它们之间没人做好的那一段：

**从看到灵感，到发起行动，到等待回应，到一起完成，到形成共同记忆。**

这就是你的价值所在。

## 5. 最终产品定位

### 5.1 一句话定位

**《今天干点啥》是一款帮助年轻用户快速决定“今天做什么”，并把这件事变成关系互动与共同记忆的小程序。**

---

### 5.2 核心用户

我建议把核心用户分为三层。

#### A 类核心用户：恋爱中的年轻情侣

这是最容易做出差异化、也是最容易形成传播的核心人群。

他们最需要的是：

1. 今天吃什么
2. 今天去哪约会
3. 周末玩什么
4. 怎样表达“我想要什么”
5. 如何获得回应
6. 如何积累共同回忆

#### B 类次核心用户：搭子型关系用户

包括：

1. 闺蜜
2. 室友
3. 同事饭搭子
4. 拍照搭子
5. 展览搭子
6. citywalk 搭子

他们未必需要恋人功能，但非常需要：

**有人一起完成一件事。**

#### C 类泛用户：独处但需要灵感的人

包括：

1. 下班后容易无聊的人
2. 周末不知道做什么的人
3. 有选择困难症的人
4. 想发内容但缺少场景的人

这类用户决定规模，但真正决定差异化的，是 A 类和 B 类。

## 6. 产品最终结构

### 6.1 一级主结构

最终建议的产品结构分成 6 大系统：

1. 首页 Today OS
2. 灵感决策中心
3. 关系协作系统
4. 内容分享社区
5. 记忆与成长系统
6. 服务与平台支撑系统

---

### 6.2 先做 3 个一级核心场景

我建议一级主场景只保留 3 个：

1. 今天吃点啥
2. 今天玩点啥
3. 今天拍点啥

原因很简单：

1. 高频
2. 好理解
3. 易分享
4. 易和恋人系统结合
5. 容易形成“决策 -> 行动 -> 分享 -> 记忆”闭环

---

### 6.3 二级增强玩法

以下方向建议保留，但不要一开始做成一级平级主频道：

1. 今天笑点啥
2. 今天聊点啥
3. 今天做点啥
4. 今天学点啥
5. 今天动点啥

更合适的方式是：

1. 作为首页卡片
2. 作为恋人/搭子互动插件
3. 作为社区活动玩法
4. 作为后续成长体系的挑战内容

这样既能保留产品扩展性，又不会导致产品主线发散。

## 7. 首页最终形态：Today OS

首页不是“功能列表页”，而应该是用户每天打开后直接能用的“今日生活面板”。

### 7.1 首页结构建议

首页建议按以下顺序布局：

1. 今日心情选择
2. 今日推荐卡组
3. 吃 / 玩 / 拍三大快捷入口
4. 恋人/搭子回应卡
5. 今日待完成
6. 附近推荐
7. 今日记忆入口
8. 今日可发一条记录

---

### 7.2 首页的核心作用

首页要解决 4 件事：

1. 今天我能做什么
2. 谁和我有关联
3. 有没有人在等我回应
4. 我今天可以完成什么、记录什么

一句话：

**首页应该像一个“今天的生活仪表盘”。**

---

## 8. 灵感决策中心：核心场景详解

## 8.1 今天吃点啥

这是当前最成熟、也最应该成为第一增长引擎的模块。

### 8.1.1 最终目标

它最终不应该只提供“随机食物”，而应该形成 6 层能力：

1. 决策层：帮用户快速决定吃什么
2. 情绪层：根据心情和状态推荐
3. 关系层：发给恋人/搭子，让对方参与
4. 服务层：连接外卖、到店、团购、菜谱
5. 内容层：形成帖子、点评、分享
6. 记忆层：记录一起吃过什么

### 8.1.2 核心功能

建议做：

1. 一键随机
2. 按情绪推荐
3. 按预算推荐
4. 按时间推荐
5. 按距离推荐
6. 按口味推荐
7. 忌口管理
8. 情侣套餐/工作餐/约会餐/一人食等场景化推荐
9. 外卖跳转
10. 到店导航
11. 收藏周末吃
12. 吃后评价
13. 一键生成吃后打卡卡片

### 8.1.3 招牌玩法

#### 玩法一：想吃你来点

链路：

1. 用户选中一个食物或店
2. 填收货信息、备注、想说的话
3. 发给恋人
4. 恋人收到卡片后点击“我请你”
5. 跳到外卖或到店服务页
6. 完成后生成“今天被投喂了”的记录

#### 玩法二：一起吃

链路：

1. 用户选中餐厅
2. 发给恋人/搭子
3. 对方回应：去 / 改一家 / 收藏 / 改时间
4. 双方确认后生成待完成卡

### 8.1.4 实现建议

1. 第一阶段以规则推荐为主，不做重 AI
2. 食物库要标准化标签：口味、价格、时段、场景、热量、情侣适配度
3. 本地服务先做跳转，不急着自己做支付履约
4. 恋人提醒用站内消息 + 订阅消息双通道
5. 吃后反馈要反哺推荐模型

## 8.2 今天玩点啥

这是第二个重点模块。

它不是旅游平台，而是：

**今天、今晚、周末半天的轻量化本地体验决策器。**

### 8.2.1 核心定位

帮助用户快速决定：

1. 今天附近玩什么
2. 下班后 2 小时能做什么
3. 预算有限适合去哪
4. 和恋人/搭子适合做什么
5. 一个人也能完成什么

### 8.2.2 内容方向

建议内容包括：

1. 咖啡店
2. 展览
3. 市集
4. 电影
5. livehouse
6. 密室
7. 桌游
8. citywalk
9. 公园露营
10. 夜游路线
11. 非遗体验
12. 手作店

### 8.2.3 核心功能

1. 按时间、预算、距离、人群筛选
2. 下雨天 / 晚上 / 情侣 / 一个人 / 多人等场景推荐
3. 地图 POI 联动
4. 官方购票/团购跳转
5. 约会发起卡
6. 共享行程卡
7. 玩后打卡

### 8.2.4 代表玩法

#### 玩法：今晚约会卡

链路：

1. 用户选中一个地点或活动
2. 发给恋人
3. 对方选择：想去 / 改时间 / 太远了 / 收藏周末
4. 双方确认后自动生成“待完成约会卡”
5. 结束后上传照片和评价

这个玩法非常适合传播，因为它天然有“约会协作”的故事性。

## 8.3 今天拍点啥

这是最容易形成漂亮内容、社区传播和视觉资产的模块。

### 8.3.1 核心定位

它不是修图工具，而是：

**给用户一个今天能拍出点东西的理由。**

### 8.3.2 核心功能

1. 今日拍摄主题
2. 今日拍照任务
3. 附近出片点
4. 情侣双人拍照任务
5. 姿势灵感卡
6. 构图灵感卡
7. 天气/时间段建议
8. 机位提示
9. 文案模板
10. 一键生成帖子模板

### 8.3.3 任务示例

1. 今天拍一张下班路上的暖光
2. 今天拍你们共同吃的那一口
3. 今天拍一张“我在等你”
4. 今天拍城市里的绿色角落
5. 今天拍一张有故事感的背影

### 8.3.4 实现建议

1. 第一阶段不做重修图
2. 核心放在“任务 + 地点 + 姿势 + 文案模板”
3. AI 可以用于生成标题、文案、滤镜建议
4. 和发现页深度联动，让拍点啥天然成为 UGC 来源

### 8.3.5 招牌玩法

#### 玩法：双人拍照任务

1. 系统给出拍照主题
2. 用户发给恋人
3. 双方一起完成
4. 自动生成双人回忆卡

---

## 8.4 二级玩法模块

### 8.4.1 今天笑点啥

建议做成首页和关系系统中的情绪缓冲层，而不是独立一级频道。

建议功能：

1. 今日一笑
2. 今日治愈卡
3. 夸夸模式
4. 情侣互逗任务
5. 深夜安抚卡

作用：

1. 提高日活
2. 提供碎片化打开理由
3. 缓解关系中的冷场

---

### 8.4.2 今天做点啥

建议做成轻任务系统，而不是待办清单。

建议功能：

1. 一个人做点啥
2. 两个人一起完成点啥
3. 30 分钟以内任务
4. 周末低成本任务
5. 约会任务
6. 关系修复任务
7. 打卡挑战

价值：

**把抽象的陪伴变成可执行动作。**

---

### 8.4.3 今天学点啥 / 今天动点啥

这两个方向可以做，但建议在第二阶段后逐步加入。

作用：

1. 学点啥：服务学生和职场新人，增强成长属性
2. 动点啥：增强健康场景与搭子互动属性

---

### 8.4.4 自定义场景

这是可以作为长期王牌的方向，但一定不要太早重投入。

### 为什么值得保留

它让产品从“官方定义场景”升级为“用户共创场景”。

例如用户可以自定义：

1. 今天遛娃啥
2. 今天养猫啥
3. 今天省钱啥
4. 今天考研啥
5. 今天送礼啥

### 建议的节奏

1. 不作为第一阶段重点
2. 在核心主线跑通后再开放
3. 先做模板化自定义，再逐步开放高级能力

这样既保留无限生长潜力，又不会让 MVP 发散。

## 9. 关系系统：产品灵魂

## 9.1 最终判断

恋人系统不能只是：

1. 绑定手机号
2. 显示头像
3. 发个消息

如果只是这样，它永远不会成为产品核心。

它必须升级成：

**关系操作层。**

---

## 9.2 关系系统的 5 层结构

### 第一层：关系建立

功能：

1. 手机号/邀请码绑定
2. 双向确认
3. 关系状态管理
4. 纪念日设置
5. 亲密昵称、头像、地址簿、偏好标签

建议：

1. 起步聚焦恋人
2. 架构上预留搭子/闺蜜/朋友模式

---

### 第二层：关系触发

你要把每个生活模块都变成“可发起关系动作”的入口。

例如：

1. 吃：想吃这个 / 一起点 / 你请我吃 / 收藏周末吃
2. 玩：今晚去这里 / 周末去这里 / 这个展你想看吗
3. 拍：今天一起拍这个 / 给我拍一张 / 我想和你拍这个
4. 做：今天一起完成这个小任务
5. 笑：今天给你送一个快乐卡

---

### 第三层：关系回应

对方不能只是看见，而必须有明确动作。

建议动作：

1. 我愿意
2. 改一下
3. 收藏以后
4. 我来安排
5. 我请你
6. 换一个吧

---

### 第四层：关系完成

行动一旦完成，要形成结构化结果：

1. 这次一起吃了什么
2. 这次一起去了哪里
3. 谁发起、谁回应、谁完成
4. 是否有照片、评价、备注

---

### 第五层：关系记忆

这是长期留存关键。

建议沉淀：

1. 一起吃过清单
2. 一起去过清单
3. 一起拍过清单

## 10. 内容社区怎么做才不会跑偏

## 10.1 发现页不做泛生活内容池

后续社区不要做成“什么都能发”的平台。

更适合的是：

**围绕真实完成行为展开的“完成后分享社区”。**

帖子最好来源于：

1. 今天吃了什么
2. 今天去了哪里
3. 今天拍到了什么
4. 今天和谁完成了什么
5. 今天收到了什么回应

这样社区就天然服务主线，而不是分散主线。

---

## 10.2 帖子模板建议标准化

建议定义 5 类模板：

1. 吃过了
2. 去过了
3. 拍到了
4. 完成了
5. 被回应了

这样做的好处：

1. 降低发帖门槛
2. 统一内容结构
3. 提升推荐效率
4. 让社区更像产品体系的一部分

---

## 10.3 社区调性：真实感大于精致感

你的社区不应该追求“小红书式完美笔记”，而应该追求：

1. 真实
2. 有温度
3. 有场景
4. 有结果
5. 有关系感

比如相比“某某店超好吃”，更适合的是：

“今天下班他给我点了这个，打开外卖的时候真的很开心。”

这类内容才是你社区真正的优势。

---

## 11. 记忆与成长系统：留存护城河

这是用户长期留存、也是最容易形成情感依赖的系统。

建议做：

1. 今日完成记录
2. 每周回顾
3. 月度生活报告
4. 共同记忆墙
5. 吃过/去过/拍过地图
6. 年度总结
7. 等级与勋章体系
8. 打卡挑战体系

这个系统的意义不是“游戏化”，而是：

**让用户看到自己和关系真的在成长。**

## 15. 风险与规避

### 15.1 不建议现在做的事

1. 一上来做太多一级频道
2. 一上来自己做完整交易闭环
3. 一上来做复杂 AI 推荐系统
4. 一上来做泛娱乐内容平台
5. 一上来围绕恋爱小游戏打转

原因：

1. 会极大分散资源
2. 会让产品主线模糊
3. 会削弱“真实生活协作”的差异点

---

### 15.2 必须提前考虑的风险

1. 内容审核与合规
2. 恋人/搭子关系中的骚扰与隐私问题
3. 图片、视频、文本安全
4. 地理位置和外部服务跳转的风控
5. 商业化过早侵入核心体验

对策：

1. 文本、图片、视频都接内容安全审核
2. 恋人系统必须双向确认、可解绑、反骚扰
3. 关系回忆默认更强隐私保护
4. 搭子系统后期必须实名认证、评价体系、安全提醒

## 18. 黑马路径总结

**情侣和搭子的日常决策神器。**

再进一步，成为：

**关系里的行动触发器。**

最后，成为：

**有共同记忆沉淀的生活系统。**

它最核心的价值不是“给用户更多选择”，而是：

1. 帮用户做决定
2. 帮用户发起行动
3. 帮用户获得回应
4. 帮用户留下记忆

这四件事一旦打通，《今天干点啥》就真正具备了成为黑马的基础。
